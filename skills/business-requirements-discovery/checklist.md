Ты выступаешь в роли эксперта по системному анализу.

Твоя задача — провести ревью документа требований по предоставленному чек-листу.

Для каждого пункта чек-листа сформируй:

- Статус
  - Пройдено
  - Частично
  - Не пройдено

- Обоснование

- Рекомендации

Если найдены проблемы, предложи конкретное исправление.

В конце сформируй итоговую оценку качества документа по шкале от 1 до 10.

# Проверочный чек-лист

## Бизнес-проблема

- [ ] Бизнес-проблема сформулирована явно.
- [ ] Понятно, кто испытывает проблему.
- [ ] Понятно, почему проблема требует решения.
- [ ] Проблема не описывает решение.

Плохо:
"Необходимо разработать систему аналитики."

Хорошо:
"Селлеры не могут оперативно получать консолидированную аналитику продаж по нескольким маркетплейсам."

---

## Бизнес-цели

- [ ] Каждая цель связана с бизнес-проблемой.
- [ ] Цели описывают результат, а не функциональность.
- [ ] Цели измеримы или потенциально измеримы.
- [ ] Нет технических целей.

Плохо:
"Реализовать Kafka."

Хорошо:
"Сократить время получения аналитических данных до 15 минут."

---

## Стейкхолдеры

- [ ] Определены основные пользователи системы.
- [ ] Определены владельцы продукта.
- [ ] Определены внешние системы или их владельцы.
- [ ] Для каждого стейкхолдера указан интерес.

---

## Бизнес-возможности

- [ ] Каждая возможность помогает достичь хотя бы одной бизнес-цели.
- [ ] Возможности описаны на бизнес-языке.
- [ ] Нет технических деталей реализации.
- [ ] Нет API, БД и инфраструктурных решений.

Плохо:
"Сервис использует PostgreSQL."

Хорошо:
"Пользователь может формировать отчет по продажам."

---

## Бизнес-требования верхнего уровня

- [ ] Требования сформулированы однозначно.
- [ ] Требования не противоречат друг другу.
- [ ] Требования проверяемы.
- [ ] Требования описывают потребность бизнеса.
- [ ] Нет проектных решений.

Плохо:
"Использовать Kafka для обработки данных."

Хорошо:
"Система должна обеспечивать обработку данных о продажах без блокировки пользовательских операций."

---

## Ограничения

- [ ] Выделены реальные ограничения.
- [ ] Ограничения не перепутаны с требованиями.
- [ ] Ограничения имеют источник (бизнес, законодательство, интеграции и т.д.).

Примеры:

- Используются данные Wildberries API.
- Поддерживаются только российские маркетплейсы.
- Отчеты должны храниться не менее 12 месяцев.

---

## Предположения

- [ ] Все недостающие данные вынесены в предположения.
- [ ] Предположения не представлены как факты.
- [ ] Для каждого предположения понятен риск.

Пример:

Предполагается, что маркетплейсы предоставляют API для выгрузки продаж.

---

## Трассируемость

- [ ] Каждая бизнес-возможность связана хотя бы с одной бизнес-целью.
- [ ] Каждое бизнес-требование поддерживает хотя бы одну бизнес-цель.
- [ ] Нет требований без бизнес-обоснования.

---

## Проверка результата AI

- [ ] AI не придумал новых стейкхолдеров без оснований.
- [ ] AI не придумал функциональность, отсутствующую во входных данных.
- [ ] Все допущения явно вынесены в раздел "Предположения".
- [ ] Требования прошли ручное ревью аналитиком.
